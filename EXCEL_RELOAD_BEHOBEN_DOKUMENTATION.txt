DATEI: Services/ReloadService.cs
─────────────────────────────────

VORHER:
  public static bool ShouldReload(string filePath)
  {
      var lastWrite = File.GetLastWriteTime(filePath);
      return lastWrite > _lastLoadTime;
  }

NACHHER:
  public static bool ShouldReload(string filePath)
  {
      var lastWrite = File.GetLastWriteTime(filePath);
      // 100ms Tolerance für Timing-Probleme
      bool needsReload = lastWrite > _lastLoadTime.AddMilliseconds(-100);
      return needsReload;
  }
  
  public static bool HasRecentChanges(string filePath)
  {
      var lastWrite = File.GetLastWriteTime(filePath);
      // Wenn letzte Änderung weniger als 5 Sekunden alt
      return (DateTime.Now - lastWrite).TotalSeconds < 5;
  }

⚙️ TECHNISCHE DETAILS:
═════════════════════════════════════════════════════════════════════════

RELOAD-SYSTEM HAT 3 EBENEN:

1. FILEWATCHER (sofort)
   - FileSystemWatcher registriert Änderungen
   - OnFileChanged Event triggt sofort
   - Schnellste Methode
   - Nur bei lokalen Datei-Änderungen

2. AUTOSYNCMANAGER (alle 2 Sekunden)
   - Timer prüft alle 2 Sekunden
   - Falls FileWatcher ausfällt → AutoSync hilft
   - Zuverlässig auch bei Netzwerk-Problemen
   - Fallback-Sicherheit

3. RELOADSERVICE (Zeitstempel-Prüfung)
   - Prüft LastWriteTime der Datei
   - Verhindert doppelte Reloads
   - Mit 100ms Tolerance
   - Sicherung gegen Race-Conditions


TIMEOUT-LOGIK:
───────────────

_lastSaveUtc = Zeit als WIR zuletzt speicherten
ReloadMaterialienAsync() wird nur unterdrückt wenn:
  (Jetzt - _lastSaveUtc) < 300ms
  
Das bedeutet:
  • Wenn WIR speichern → 300ms Ruhe (verhindert Race-Conditions)
  • Wenn ANDERE speichern → wird sofort geladen!
  • Nach 300ms → normale Reload-Logik
